# Leanforge v1.6.4

### 입력은 의도. _출력은 검증된 소프트웨어._

Claude Code & Codex를 위한 **플러그인 하네스**: 모호한 입력을 검토된 3-doc으로 만들고, 증거 게이트로 실행하며, 다음 에이전트가 읽을 프로젝트 기억을 남긴다.

**추측이 남지 않을 때까지 묻는다 — 말한 적 없는 결정까지 찾아서.**<br/>
**천장이 아니라 바닥 — 더 나은 모델이 곧 더 나은 결과다.**<br/>
**세션보다 오래 사는 기억 — 결정과 이유가 프로젝트의 문서 체계에 남는다.**<br/>
**완료는 증거다 — 종료 코드와 실제 응답. "된 것 같다"는 통과하지 못한다.**<br/>
**중단된 실행은 로컬 Leanforge 상태에서 복구한다.**

## 모든 프롬프트는 과소명세다

"예약 시스템을 만들어 줘"는 요구사항처럼 읽힌다. 실제로는 목표 선언이다
— 요구사항 수준의 결정은 전부 열려 있다: 예약과 시술의 카디널리티,
시술이 폐지될 때 기존 예약의 생명주기, 취소가 슬롯을 즉시 해제하는지
보류하는지.

에이전트는 그 공백 앞에서 멈추지 않는다. 각각을 그럴듯한 디폴트로 그
자리에서 메우고, 컴파일되고 실행되고 데모까지 매끈한 코드를 내놓는다.
*한 예약 = 시술 하나*는 이제 하중을 받는 스키마다 — 뒤에 어떤 결정
기록도 없이. 비용은 몇 달 뒤, 패키지 시술이 기능이 되는 날
마이그레이션으로 청구된다.

**프롬프트는 의도를 담는다. 구현은 결정으로 고정된다.** 감독 없는
에이전트는 당신이 본 적 없는 결정을 전부 내리고, 그중 무엇도 보고하지
않는다.

둘째 실패가 첫째를 증폭한다: **아무것도 지속되지 않는다.** 내려진 결정은
세션 트랜스크립트에 살고, 트랜스크립트는 세션과 함께 죽는다. 다음 세션은
코드만으로 프로젝트 상태를 재도출하는데, 코드는 결과를 기록할 뿐 이유를
기록하지 못한다. 귀결은 구조적 표류다 — 정리된 질문이 다시 논쟁되고,
불변식이 서로 호환되지 않게 재구현되고, 관례가 모듈마다 갈라진다.

Leanforge는 두 지점에 개입한다: 열린 결정은 **코드가 존재하기 전에** 전수
열거되고, 모든 처분은 그 이유와 함께 **모든 미래 세션이 가장 먼저 읽는
경로에** 기록된다.

## 네 가지 명령: 설계, 실행, 확장

```
  /leanforge:prime <입력> ──▶  /leanforge:run  ──▶  동작하는 코드 + 프로젝트 하네스
                              ├─ /leanforge:set (1회성 온보딩)
                              └─ /leanforge:harness (워크플로 설계)
```

| 명령 | 입력 | 산출 |
|---|---|---|
| `Leanforge:Prime` (`/leanforge:prime`) | 무엇이든 — 한 줄부터 문서 뭉치까지 | 검토를 거친, 실행 가능한 설계 계약 |
| `Leanforge:Run` (`/leanforge:run`) | 승인된 계약 | 검증된 코드 + 프로젝트 하네스 |
| `Leanforge:Set` (`/leanforge:set`) | 기존 코드베이스 | 프로젝트 하네스 (1회성) |
| `Leanforge:Harness` (`/leanforge:harness`) | 설계 브리프, 스펙, 커스텀 에이전트 | 검증된 agent workflow + 스킬 스펙 |

## `/leanforge:prime` — 의도에서 실행 가능한 명세로

`/leanforge:prime`는 임의의 입력을 받는다 — 한 줄짜리 아이디어, 다른 도구가 생성한
요구사항 문서, 흩어진 메모, 혹은 감 하나. 전부 같은 지위로 들어온다:
**도전받는 재료이지, 확정된 진실이 아니다.** 문서가 존재한다는 사실은 그
뒤에 설계 대화가 있었다는 증거가 아니다. 입력 내부의 충돌은 침묵의
선택이 아니라 질문이 되고, 섞여 든 코드 조각은 구현으로 승계되는 대신
그것이 인코딩한 행위 계약 — 입력, 출력, 불변식 — 으로 환원된다.

그 재료로부터 `/leanforge:prime`는 설계가 답할 의무가 있는 결정을 — 말했든 말하지
않았든 — 전수 열거하고, 각각을 두 종결 상태 중 하나로 몰아간다: **표명된
의도에서 도출되거나, 질문되거나.** 침묵의 디폴트는 종결 상태가 아니다.

질문량은 구조적으로 제한된다:

- 이전 답에서 도출 가능하면 → 묻지 않는다.
- 확정된 메커니즘 내부의 튜닝값이면 → 디폴트로 두고, 조정 가능 항목으로
  기록한다.
- 모든 질문은 추천을 앞세운다 — 수락은 키 입력 한 번이다.
- 도메인 질문에는 *"이 중에 없음"* 선지가 항상 열려 있다 — 도메인 지식의
  권위는 선택지 목록이 아니라 당신에게 있다.

프로젝트의 첫 사이클에서 `/leanforge:prime`는 본격적인 빌드에 선행해야 할 대화를
추가로 수행한다: 이 프로젝트가 무엇이고 무엇이 아닌지, 그 배후의 도메인
모델 — 엔티티, 생명주기, 규칙 — 그리고 스택. 가져온 스택이 없으면
트레이드오프를 명시해 추천한다. 어떤 기술 방향도 침묵으로 확정되지
않는다.

결과가 당신에게 도달하기 전, 저작에 참여하지 않은 별도 에이전트의 독립
검토를 거친다 — 당신이 말한 것이 누락되거나 왜곡되지 않았는지, 산출물이
쓰인 그대로 실행 가능한지. 이를 최종으로 만드는 유일한 사건은 당신의
승인이다.

## `/leanforge:run` — 증거로만 통과하는 실행

`/leanforge:run`는 승인된 계약을 이어받아 그 시점부터 git 상태 전체를 소유한다.

**스케줄링.** plan은 명시적 의존성 그래프를 담는다. `/leanforge:run`는 어떤 git
변경도 가하기 전에 그래프를 검증한다 — 순환, 끊어진 참조, 커버리지 공백.
결함 있는 그래프는 실행 시점에 때울 대상이 아니라 생산자 결함으로서
조기에 실패한다. 독립 task들은 동시에 실행되고, 구현자 각각이 격리된 git
worktree를 받으며, 브랜치가 실제로 전진했는지와 diff가 선언된 표면을
건드리는지 검증하는 병합 게이트를 통해 복귀한다. 구현자의 자기 보고는
무게를 갖지 않는다.

**검증은 위험에 비례한다.** 기계적 rename과 결제 경로 변경이 같은 절차를
받지 않는다. 고위험 task는 spec 슬라이스와 raw diff에 대한 독립 리뷰를
받는다 — 구현자의 요약은 전달되지 않는다. 그것이 리뷰어가 앵커링되는
경로이기 때문이다.

**게이트, 끝에서 끝까지.**

- 각 병렬 wave는 병합된 base에 대한 프로젝트 검증 스위트 실행으로 끝난다
  — task 간 상호작용이 최초로 존재하게 되는 지점이다.
- 완료 시점에는 전체 검증이 재실행되고 **런타임 smoke**가 더해진다:
  spec이 선언한 서비스는 실제로 부팅되어 실요청에 응답해야 한다.
  *컴파일된다*와 *동작한다*는 다른 명제다.
- 평가에 도달하지 못한 검증 — 단언 전에 명령이 죽은 경우 — 은 추론된
  통과가 아니라 실패다.

**Escalation은 동기적이다.** 막힌 task는 멈춰서 당신의 답을 기다린다.
답을 가정하고 그 위에 쌓지 않는다.

**git은 당신의 것이다.** 기존 프로젝트는 feature branch에서 실행된다.
main은 직접 쓰이지 않으며, 최종 통합 — merge, PR, 수동 — 은 결코
자율적으로 일어나지 않는다. 더러운 working tree나 push되지 않은 커밋은
실행을 시작 전에 중단시킨다.

전부 통과하면 `/leanforge:run`는 프로젝트 하네스를 생성 또는 갱신하고, 변경 전체에
대한 최종 독립 리뷰를 한 번 더 돌린 뒤, 설계 계약을 `.leanforge/` 아래에
아카이브한다. 승인이나 아카이브 전에 실행이 끊기면 `.leanforge/run.json`이
거친 복구 상태를 보존해 다음 에이전트가 재개 또는 폐기를 명시적으로 고를 수 있다.

## `/leanforge:set` — 기존 코드베이스 온보딩

1회성 전환이다 — task 실행기가 아니다. `/leanforge:set`은 먼저 코드베이스를
스캔한 뒤, 코드가 증언할 수 없는 것을 정확히 캐낸다: 코드 경로는 auth
체크가 무엇을 *하는지*를 보여줄 뿐, 그것이 *정책의 전부인지*는 보여주지
못한다. Elicitation은 위험 가중이다 — 추론 가능하고 틀려도 싼 것은
추론하고, 추론 가능하지만 틀리면 비싼 것 — 도메인 불변식, 보안 경계,
체크 뒤의 비즈니스 모델 — 은 당신에게 확인받는다.

질문은 평이한 언어로 도착한다 — *"이게 바뀌면 저것도 같이 바뀌어야
합니까?"* — 그리고 답은 정밀한 규칙으로 다시 컴파일된다. 하네스 전체를
생성하고, 커밋은 당신에게 남기고, 종료한다. 그 뒤로 프로젝트는
`/leanforge:prime` → `/leanforge:run`로 돈다.

## 번들 지원 스킬

### `/leanforge:harness` — 워크플로 설계 & 팀 명세

harness 스킬은 팀 스펙, 역할 브리프, specialist skill, custom-agent helper,
검증 루프, benchmark pack까지 포함한 repo-local agent workflow를 설계·점검·수리·검증한다.

`/leanforge:run`가 작성하는 per-cycle 하네스(`CLAUDE.md` / `AGENTS.md` / `docs/`)의 상위 설계 레이어다.
`/leanforge:run`가 실행-시점 오케스트레이션을 맡는다면, `/leanforge:harness`는 재사용 가능한 워크플로 패턴을
설계하는 역할이다 — 프로젝트를 넘나드는 에이전트, 규모에 맞는 팀 관례, 병렬 실행에
삽입되는 specialist 역할.

`/leanforge:harness`가 필요한 경우:

- 대규모 실행 전에 커스텀 multi-agent 워크플로를 설계하고 테스트하고 싶을 때
- 팀 표준, 도메인 지식, 검증 규칙을 실행 가능한 스펙으로 고정하고 싶을 때
- 기존 워크플로의 안전성, 일관성, 정확성을 감사하고 싶을 때
- 다른 에이전트나 프로젝트가 호출하는 스킬을 만들고 싶을 때

## 한 사이클의 해부

`/leanforge:prime` → `/leanforge:run`는 승인 지점이 정확히 두 개인 단일 파이프라인이다 — 둘
다 당신의 것이다. 그 사이의 전부는 자율로 돈다.

```
/leanforge:prime 입력 분해 → 모든 열린 결정의 처분 → 계약 작성
          → 독립 검토 → ▶ 당신의 승인

/leanforge:run   그래프 검증 → 병렬 wave 실행 → 통합 게이트
          → 런타임 smoke → 하네스 생성/갱신 → 최종 리뷰
          → ▶ 당신의 승인 → 계약 아카이브
```

`/leanforge:prime`가 디스크에 남기는 것은 `.leanforge/`의 3문서 **설계 계약**이다:

- **spec** — *무엇*의 권위: 행위 규칙, 불변식, API 표면, 명시적 처분이
  달린 모든 엣지 케이스, 그리고 결과가 통과해야 할 검증.
- **plan** — *어떻게*의 청사진: task별 행위 계약과 `/leanforge:run`가 스케줄링하는
  의존성 그래프.
- **handoff** — 관장 문서: 세 문서의 관계와, 어떤 단계도 넘을 수 없는
  하드 게이트.

계약은 구조적으로 자체 완결이다 — 그것을 만든 대화 없이도 미래의
에이전트가 행동할 수 있도록 쓰인다. 코드에서 재도출할 수 없는 결정은 그
추론을 본문에 지닌다. 권위 위계도 명시적이다: spec과 코드가 어긋나면
spec이 이긴다. spec 자체가 틀려 보이면 에이전트는 그것을 고치지 않는다 —
당신에게 돌아온다.

`/leanforge:run`가 완료되면 계약은 `.leanforge/` 아래 사이클 단위로 아카이브된다 —
무엇이, 언제, 왜 결정됐는지의 영속 기록이다. 진행 중 작업은 루트
`.leanforge/{handoff,spec,plan}.md`로 드러나고, 중단된 실행은 `.leanforge/run.json`으로 보호된다.

Legacy 상태 호환성: Leanforge rename 이전에 만든 저장소에는 `.dryforge/`가 남아 있을 수 있다.
Leanforge의 canonical 상태 디렉터리는 `.leanforge/`다. `.leanforge/`가 없고 `.dryforge/`만 있으며
active `run.json`, 루트 3문서, `worktrees/`가 없을 때만 `Run`/`Set`이 `.leanforge/`로 이동하고
`.leanforge/migration.json`을 기록한다. 진행 중인 legacy run은 자동 migration하지 않는다 —
먼저 완료, 재개, 또는 폐기해야 한다.

## 지속되는 것 — 프로젝트 하네스

```
your-project/
├── CLAUDE.md                  # Claude Code 진입점 — 정체성 + 작업 규칙
├── AGENTS.md                  # Codex 진입점 — 동일 내용
├── docs/
│   ├── architecture.md        # 구성: 컴포넌트, 흐름, 의존성
│   ├── business-rules.md      # 도메인 로직: 엔티티, 불변식, 엣지 케이스
│   ├── security.md            # 정책: 보호 자산, 접근, 감사
│   ├── standards.md           # 규칙: 하드 게이트, 관례, 경계
│   ├── engineering-notes.md   # 체득 지식: 함정, 메커니즘, 체크리스트
│   ├── operations.md          # 구동법: 셋업, 빌드, 배포
│   ├── contracts.md           # 외부 인터페이스 계약
│   ├── harness/               # (선택) 재사용 워크플로, 팀 스펙, 역할 브리프
│   └── tracking/
│       ├── status.md          # 전체 scope 대비 현재 위치
│       ├── decisions/         # 결정 기록 — 무엇을, 왜 선택했는가
│       └── findings.md        # 미해결로 알려진 문제
└── <module>/AGENTS.md         # 모듈별 범위, 경계, 불변식
```

- **세션 독립적 맥락.** 새 대화는 하네스를 읽고 아키텍처·규칙·이유를
  장전한 채 재개한다. 재설명은 워크플로의 일부가 아니다.
- **이유가 1급이다.** 무엇을 선택했는가에 더해 *왜* — 의도가 조용히
  뒤집히지 않고, 정리된 논쟁은 정리된 채로 남는다.
- **덧붙이지 않고 유지한다.** 갱신은 양방향으로 정합한다: 기존 서술을
  무효화하는 작업은 그 서술을 고친다. 하네스는 현실을 추적하거나,
  고쳐진다.
- **락인 제로.** 표준 `CLAUDE.md` / `AGENTS.md`다. 생성된 문서에는
  Leanforge 어휘도 전용 포맷도 없다 — 어떤 에이전트든 읽는다. 내일
  Leanforge를 지워도, 자산은 남는다.

## 어디에 서 있는가

계획이나 오케스트레이션을 건드리는 도구는 많다. 차이는 각자가 무엇을
신뢰하는가에 있다.

- **맨 에이전트 대비** — 기준점 없는 강한 모델은 매 세션 전부를
  재도출하고, 모델이 강할수록 더 넓게 표류한다. Leanforge는 같은 결정과
  같은 이유를 매 세션 공급한다.
- **스펙 생성기·plan 모드 대비** — 그들은 당신이 말한 것을 정리한다.
  Leanforge는 당신이 말하지 *않은* 것을 열거한다 — 그리고 들어온 문서를
  포맷팅하는 대신 도전한다.
- **처방형 워크플로 대비** — "X면 Y" 규칙집은 영원히 같은 절차 비용을
  물리고, 품질을 규칙 저자의 예지력에 상한 짓는다. Leanforge는 계약과
  바닥만 고정하고 천장을 연다 — 모델 업그레이드가 곧 산출물 업그레이드로
  번역된다.
- **병렬 오케스트레이터 대비** — 의도 grounding 없는 병렬성은 잘못된 것을
  더 빨리 출하한다. Leanforge는 승인된 spec의 하류에서만 병렬화하고,
  게이트를 통과한 것만 병합한다.
- **상태 대시보드 대비** — 많은 도구가 실행 후 이벤트를 추적한다. Leanforge는
  *결정*을 추적한다 — 추론, 열린 질문, 강제된 게이트. 실행 로그는 무슨 일이
  있었는지 말하고, 설계 계약은 다음 에이전트가 의도를 다시 도출하게 한다.

## 철창이 아니라, 나침반

- **천장이 아니라 바닥.** Leanforge는 인터페이스 계약과 절차 골격을
  고정한다. 결론은 모델의 몫으로 남는다. 처방적 규칙은 도구를 저자의
  예지력에 상한 짓는다 — 바닥만 깔면, 더 나은 모델이 자동으로 더 나은
  결과를 낸다.
- **경계 지어진 자율.** 당신이 의도를 승인하면, 그 경계 안에서 에이전트는
  자유롭게 판단한다. 자율은 승인된 의도의 실행이다 — 의도를 스스로 정하는
  일이 아니다.
- **추측하지 않고 묻는다.** 의도에서 도출되지 않는 것은 질문으로 돌아온다
  — 동기적으로, 진행하기 전에. 추측 한 번이 멈춤 한 번보다 비싸다.

## 플랫폼 노트

- **명시 호출 전용.** 핵심 lifecycle과 harness는 자동으로 트리거되지 않는다. 부르지 않으면
  아무것도 돌지 않는다.
- **단일 소스, 두 플랫폼.** Claude Code와 Codex 산출물은 플랫폼 중립의
  canonical `src/skills/` 트리에서 committed `claude/` 및 `codex/plugin/`
  bundle로 빌드된다. 빌드는 릴리스 전에 plugin version parity와 공유
  reference file parity를 검사한다.
- **지원 스킬은 플러그인에 포함된다.** `/leanforge:harness`는 워크플로 설계, 감사,
  수리, 재사용 가능한 specialist skill 작성을 위해 플러그인과 함께 설치된다.
  lifecycle의 중심은 계속 `Prime`, `Run`, `Set`이다.
- **요구 사항.** `git`, 그리고 Claude Code 또는 Codex.

---

## 언제 쓰지 말아야 하는가

- **한 줄 수정에는 설계 대화가 필요 없다.** Leanforge는 재작업을 없애기
  위해 비용 — elicitation, 검증 — 을 선불한다. 그 거래는 기능과
  프로젝트에서 남고, 오타 수정에서는 남지 않는다.
- **git이 필수다.** 실행 규율이 브랜치와 worktree 위에 세워져 있다.
- 질문에 답하는 시간은 실행에서 회수된다 — 그리고 당신에게 묻는 대신
  하네스를 읽는 이후의 모든 사이클에서 다시 회수된다.

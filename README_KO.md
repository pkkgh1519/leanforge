## 가두지 않아요. 방향만 잡죠.

같은 모델이라도 결과는 더 좋아질 수 있어요. dryforge는 일일이 시키는 대신, 잘 판단할 바탕을 깔고 탈선만 막아 — 모델이 가진 힘을 끝까지 끌어내죠. **천장이 아니라 바닥(floor)**을 설정해서, 더 좋은 모델일수록 결과도 더 좋게요.

- **규칙 대신, 판단 기준을 줘요.** '이럴 땐 이렇게' 같은 규칙으로 모델을 가두지 않아요. 대신 스스로 옳게 판단할 기준을 깔아주죠. 규칙은 모델을 묶지만, 좋은 기준은 모델을 풀어주거든요.
- **모델의 잠재력을 깨워요.** 많은 도구는 'AI는 못 미더우니까'라며 절차로 옭아매다 모델의 실력까지 눌러요. dryforge는 천장을 막지 않으니, 모델이 가진 최고치가 그대로 나오죠. 더 좋은 모델일수록 결과도 더 좋고요.
- **가두지 않고, 경계만 그어요.** 당신이 의도를 승인하면, 그 안에서 에이전트는 자유롭게 판단해요. 통제가 아니라 경계예요(*bounded autonomy*).
- **'된 것 같다'는 안 통해요.** 완성은 에이전트의 느낌이 아니라 실제로 돌려본 증거로 판정해요. 그것도 구멍을 찾는 게 임무인 독립된 적대적 시각으로요.
- **위험한 만큼만 검증해요.** 모든 일에 똑같은 절차를 강요하지 않아요. 위험한 건 두껍게, 가벼운 건 가볍게 — 딱 필요한 만큼만.
- **모르면 추측 대신 물어요.** 의도에서 안 나오는 건 멋대로 정하지 않고 당신에게 물어요. 추측 한 번이 멈춤보다 나쁘니까요.

## 프로젝트가 체계를 가져요

dryforge가 남기는 건 코드만이 아니에요. 프로젝트의 구조·규칙·결정과 그 **이유**를 실제 문서로 남겨, 지속되는 체계를 세우죠. 이 살아있는 문서 묶음이 바로 **프로젝트 하네스**예요.

```
your-project/
├── CLAUDE.md                  # Claude Code 진입점 — 정체성 + 작업 규칙
├── AGENTS.md                  # Codex 진입점 — 같은 내용
├── docs/
│   ├── architecture.md        # 구조: 구성요소·흐름·의존성
│   ├── business-rules.md      # 도메인 규칙: 엔티티·불변식·엣지 케이스
│   ├── security.md            # 보안 정책: 보호 자산·접근·감사
│   ├── standards.md           # 규칙: 하드 게이트·관례·경계
│   ├── engineering-notes.md   # 노하우: 함정·메커니즘·체크리스트
│   ├── operations.md          # 운영: 셋업·빌드·배포
│   ├── contracts.md           # 외부 인터페이스 계약
│   └── tracking/
│       ├── status.md          # 전체 범위 대비 현재 위치
│       ├── decisions/         # 결정 기록 — 무엇을, 왜
│       └── findings.md        # 미해결 문제
└── <module>/AGENTS.md         # 모듈별 범위·경계·불변식
```

- **세션이 바뀌어도, 맥락은 그대로예요.** 새 세션도 처음부터 다시 설명할 필요 없이, 하네스를 먼저 읽고 프로젝트를 다 아는 채로 이어가요.
- **결정만이 아니라, '왜'까지 남아요.** 무엇을 정했는지에 더해 왜 그렇게 했는지가 함께 기록돼요. 그래서 나중에 의도가 뒤집히거나 같은 고민을 반복하지 않죠.
- **코드와 따로 놀지 않아요.** 작업할 때마다 갱신되고, 실제 코드와 어긋나면 맞춰져요. 문서가 현실과 어긋나 쓸모없어지는 일이 없어요.
- **dryforge 없이도 통해요.** 하네스는 `CLAUDE.md`·`AGENTS.md` 같은 표준 파일이라, 다른 AI 도구로 열어도 그대로 읽혀요. 어디에도 묶이지 않죠.

## 같은 프롬프트, 다른 코드

그 '같은 맥락'이 왜 그렇게 중요할까요? 같은 프롬프트도 세션이 다르면 다른 코드를 뱉어요. AI가 부족해서가 아니에요 — 오히려 너무 똑똑해서, 참고할 맥락이 없으면 매번 스스로 새로 추론하거든요. 똑똑할수록 편차는 더 커지고요. 사람도 어제와 오늘 생각이 다르잖아요.

<table>
<tr>
<td><sub>세션 1</sub>

```js
function sortUsers(list) {
  return list.sort((a, b) =>
    a.name.localeCompare(b.name))
}
```
</td>
<td><sub>세션 2</sub>

```js
const sorted = [...users].sort(
  (a, b) => a.name > b.name ? 1 : -1
)
```
</td>
</tr>
</table>

답은 단순해요 — 강한 모델에게 매번 **같은 맥락**을 쥐여주는 것. 그게 바로 하네스이고, 그게 dryforge예요.

## 어떻게 시작하든, `go`가 끝내줘요

상황에 맞게 고르세요. **`ready`**와 **`set`**은 시작 방법이 다를 뿐, 둘 다 **`go`**로 이어져요. 이미 만든 프로젝트라면 **`migration`**으로 들여오고요.

```
  ready ─┐
         ├──▶  go  ──▶  결과물 + 프로젝트 하네스
  set  ──┘

  migration  ──▶  기존 프로젝트를, dryforge 안으로
```

| | 명령어 | 하는 일 |
|---|---|---|
| **아이디어만 있어요** | `/dryforge:ready` | 막연한 생각을 콕콕 물어, 모호함이 없을 때까지 캐내 완전한 설계로 만들어줘요. 기술 스택이 없으면 추천하고, 프로젝트 규모에 딱 맞게. |
| **정리해둔 게 있어요** | `/dryforge:set` | 메모든 다른 도구가 만든 문서든 가져오세요. `set`이 읽고, 빠진 걸 짚어 묻고, 실제로 만들 수 있게 다듬어요. |
| **→ 그리고 둘 다 모여요** | `/dryforge:go` | 계획대로 낭비 없이 만들어요. 미리 계산한 순서로 여러 작업을 동시에, 위험한 부분은 더 꼼꼼히 검증해요. → 결과물 **+** 프로젝트 하네스. |
| **이미 만든 프로젝트예요** | `/dryforge:migration` | 기존 코드를 읽어 '프로젝트 하네스'를 자동으로 만들어줘요 — 적어두지 않은 의도까지 끄집어내서요. 처음부터 다시 안 해도, 한 번에 dryforge 안으로. |

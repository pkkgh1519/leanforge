# dryforge

*[English](./README.md)*

> 만들고 싶은 걸 말하면, 명령어 두 개로 끝.

```
/plugin marketplace add fn-opt/dryforge
/plugin install dryforge
```

## 어디서 시작하든, 결과는 같습니다

시작점이 뭐든 dryforge가 제대로 된 코드로 만들어냅니다.

### 경로 1 — 아이디어에서 시작

```
/dryforge:ready    →    /dryforge:go
```

아이디어는 있는데 문서는 없을 때. `ready`가 대화로 의도를 끌어냅니다 —
중요한 건 깊게 물어보고, 코드 보면 알 수 있는 건 알아서 처리하고 —
spec과 plan을 직접 씁니다. 그다음 `go`가 만듭니다.

### 경로 2 — 문서에서 시작

```
/dryforge:set    →    /dryforge:go
```

spec이랑 plan이 이미 있을 때. 직접 쓴 거든, 다른 도구로 뽑은 거든.
경로가 틀리고, 빠진 게 있고, 실제 코드랑 안 맞는 내용이 있을 수 있죠.
`set`이 실제 코드에 맞춰 검증하고, 잘못된 건 고치고, 없는 건 채우고,
실행 가능한 상태로 만들어줍니다. 그다음 `go`가 만듭니다.

## 진행 과정 미리보기

```
나:  /dryforge:ready 알림 기능을 만들고 싶어요. 내 글에 답글 달리면 알려주세요.

      dryforge가 코드를 읽고, 질문합니다.
      체크리스트가 아니라 대화입니다. 중요한 것만 물어봅니다.

      "알림을 실시간으로 보낼까요, 모아서 보낼까요?
       실시간이면 웹소켓이 필요해요. 모아서 보내는 건 지금 스택으로
       충분하고, 주기적으로 다이제스트 메일을 보내는 방식이에요.
       사용자 입장에서 어느 쪽이 맞나요?"

      답하면 spec과 plan을 씁니다.

나:  /dryforge:go

      최대 8개 작업이 각자 격리된 git worktree에서 동시에 돌아갑니다.
      작업마다 spec 리뷰. wave마다 통합 검증. 합친 코드에 코드 리뷰.

      애매하면 추측해서 엉뚱한 걸 만드는 대신, 물어봅니다.
```

## 기존 플러그인과 어떤 점이 다른가요?

에이전트한테 큰 작업을 시켜본 적 있으면 아실 겁니다:

- 뭘 만들지 제대로 파악도 안 하고 코딩부터 합니다. 중간에야
  "아 이것도 정해야 했네" 하고 깨달아요.
- 뭘 하든 같은 절차를 밟습니다. 설정 파일 하나 고치는 것도 TDD,
  한 줄 바꾸는 것도 3단 리뷰. 작업에 맞추는 게 아니라 절차를 따르는 거예요.
- plan에 코드를 잔뜩 써놓는데, 실행할 때쯤이면 이미 안 맞아서
  에이전트가 자기 plan이랑 싸웁니다.
- 동시에 돌릴 수 있는 작업이 순서대로 하나씩 돌아갑니다.
  뭐가 뭘 기다려야 하는지 아무도 계산 안 했으니까요.

dryforge는 두 단계로 나눠서 해결합니다:

**만들기 전에** — 뭘 만들지 먼저 파악합니다.
깊게 묻고, spec 쓰고, 의존성 계산하고, 뭘 동시에 돌릴 수 있는지 정합니다.

**만들 때** — 새 세션에서, plan대로 만듭니다.
격리된 worktree에서 병렬 실행. 작업에 맞는 검증.
애매하면 넘겨짚지 않고 물어봅니다.

spec이 기준입니다. 전부 거기서 시작됩니다.

## 명령어

### `/dryforge:ready <goal>`

아이디어를 던지면 시작. 코드를 읽고, 중요한 질문을 던지고,
spec과 plan을 직접 씁니다. 설계 결정은 깊게 파고들고,
코드 보면 알 수 있는 건 알아서 처리합니다.

### `/dryforge:set <spec> <plan>`

문서를 던지면 시작. 코드를 읽고 틀린 거 찾아냅니다 — 경로가 안 맞고,
빠진 게 있고, 코드랑 안 맞는 plan. 고치고, 의존성 계산하고,
실행 가능한 상태로 만들어줍니다.

### `/dryforge:go`

만듭니다. 새 세션에서 (`/clear`) 실행하세요.
의존성 그래프를 wave로 나누고, 격리된 git worktree에서
병렬 에이전트를 띄우고, spec 리뷰와 통합 검증을 거치고,
다 끝나면 main에 머지할지 물어봅니다.

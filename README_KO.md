Leanforge는 Claude Code와 Codex를 위한 플러그인 하네스입니다. 모호한 요청을 바로 코드로 밀어붙이지 않고, 먼저 검토 가능한 실행 계약으로 정리한 뒤 증거를 남기며 구현합니다. 다음 에이전트가 이어받을 수 있는 프로젝트 맥락도 함께 남깁니다.

## 60초 시작

### 설치

**Claude Code**

```
/plugin marketplace add pkkgh1519/leanforge
/plugin install leanforge
```

**Codex**

```
codex plugin marketplace add pkkgh1519/leanforge
codex plugin add leanforge@leanforge
```

<sub>`git`과 Claude Code 또는 Codex가 필요합니다.</sub>

<sub>배포: marketplace source는 `pkkgh1519/leanforge`, 설치되는 plugin identity는 `leanforge`, 사용자에게 보이는 제품명은 Leanforge입니다.</sub>

### 설치 확인

설치가 끝나면 명령 팔레트에서 다음 명령을 확인하세요.

- `Leanforge:Prime` (`/leanforge:prime`)
- `Leanforge:Run` (`/leanforge:run`)
- `Leanforge:Set` (`/leanforge:set`)
- Codex 전용: `Leanforge:Run TDD` (`/leanforge:run-tdd`)

### 첫 실행

1. 작고 결과를 확인하기 쉬운 요청으로 시작합니다.

   ```text
   /leanforge:prime 1인 서비스 사업자를 위한 최소 예약 흐름을 만들어 줘.
   ```

2. 생성된 spec, plan, handoff를 검토하고 승인합니다.
3. 승인한 계약을 실행합니다.

   ```text
   /leanforge:run
   ```

4. 마지막 요약에서 변경된 파일, 실행한 검증, 확인된 증거, 필요한 프로젝트 하네스 업데이트를 확인합니다.

### 이런 경우에 쓰세요

- 요청이 애매해서 에이전트가 마음대로 기본값을 정하면 나중에 비용이 커질 때.
- 여러 파일, 여러 에이전트, 여러 단계가 하나의 실행 계약 아래 움직여야 할 때.
- 결정과 검증 결과가 지금 대화가 끝난 뒤에도 남아야 할 때.
- 한 번 실행하고 끝나는 agent run보다, 중간에 멈춰도 이어갈 수 있는 로컬 워크플로가 필요할 때.

## 모든 프롬프트에는 빈칸이 있습니다

"예약 시스템을 만들어 줘"는 요구사항처럼 보이지만, 실제로는 목표에 가깝습니다. 예약 하나가 시술 하나만 가리키는지, 시술이 사라지면 기존 예약은 어떻게 되는지, 취소하면 슬롯을 바로 풀어도 되는지 같은 결정은 아직 비어 있습니다.

에이전트는 보통 그 빈칸 앞에서 멈추지 않습니다. 그럴듯한 기본값을 고르고, 컴파일되고 데모도 되는 코드를 만듭니다. 문제는 그 순간 선택된 기본값이 스키마와 API에 박힌다는 점입니다. *한 예약 = 시술 하나*라는 가정은 나중에 패키지 시술이 필요해지는 순간 마이그레이션 비용으로 돌아옵니다.

**프롬프트는 의도를 담고, 구현은 결정으로 굳어집니다.** 감독 없이 달리는 에이전트는 사용자가 보지 못한 결정을 계속 내리지만, 그 결정을 따로 보고하지는 않습니다.

문제는 거기서 끝나지 않습니다. 결정이 세션 안에만 남으면 다음 세션은 다시 코드만 보고 프로젝트를 추론합니다. 코드는 결과를 보여주지만 이유를 설명하지 않습니다. 그래서 이미 정리된 질문이 다시 열리고, 불변식이 모듈마다 다르게 구현되고, 관례가 조금씩 갈라집니다.

Leanforge는 이 두 지점에 개입합니다. 코드를 쓰기 전에 열려 있는 결정을 드러내고, 선택한 이유를 다음 세션이 가장 먼저 읽는 위치에 남깁니다.

## 세 가지 명령: 설계, 실행, 온보딩

```
  /leanforge:prime <입력> ──▶  /leanforge:run  ──▶  동작하는 코드 + 프로젝트 하네스
                              └─ /leanforge:set (1회성 온보딩)
```

| 명령 | 입력 | 산출 |
|---|---|---|
| `Leanforge:Prime` (`/leanforge:prime`) | 한 줄 아이디어부터 긴 문서까지 | 검토를 거친 실행 계약 |
| `Leanforge:Run` (`/leanforge:run`) | 승인된 계약 | 검증된 코드와 프로젝트 하네스 |
| `Leanforge:Set` (`/leanforge:set`) | 기존 코드베이스 | 프로젝트 하네스 초기화 |

## `/leanforge:prime` — 의도를 실행 가능한 계약으로 바꾸기

`/leanforge:prime`는 입력의 형태를 가리지 않습니다. 한 줄짜리 아이디어, 다른 도구가 만든 요구사항 문서, 흩어진 메모, 아직 감뿐인 방향성까지 모두 받을 수 있습니다. 다만 들어온 내용을 그대로 믿지는 않습니다. 문서가 있다는 사실만으로 설계 대화가 끝났다고 보지 않습니다.

입력 안의 충돌은 조용히 골라잡지 않고 질문으로 올립니다. 코드 조각이 섞여 있어도 구현을 그대로 가져가지 않습니다. 그 코드가 뜻하는 동작, 입력과 출력, 불변식만 계약으로 추립니다.

그다음 Leanforge는 설계가 반드시 답해야 할 결정을 찾습니다. 사용자가 말한 것도, 말하지 않았지만 구현 전에 정해야 하는 것도 포함합니다. 각 결정은 두 상태 중 하나로 닫혀야 합니다. 사용자의 의도에서 도출되었거나, 사용자에게 질문되어야 합니다. 아무 말 없이 기본값을 고르는 것은 완료가 아닙니다.

질문은 무작정 늘리지 않습니다.

- 이전 답에서 도출할 수 있으면 묻지 않습니다.
- 확정된 메커니즘 안의 튜닝값이면 기본값으로 두되, 나중에 조정할 수 있게 기록합니다.
- 질문마다 추천안을 먼저 제시해, 동의할 때는 한 번에 넘길 수 있게 합니다.
- 도메인 질문에는 항상 *"이 중에 없음"*을 열어 둡니다. 도메인 지식의 최종 권한은 선택지가 아니라 사용자에게 있습니다.

프로젝트의 첫 사이클에서는 빌드에 앞서 프로젝트 자체도 정리합니다. 이 프로젝트가 무엇이고 무엇이 아닌지, 핵심 엔티티와 생명주기, 규칙, 스택을 확인합니다. 이미 정해진 스택이 없으면 트레이드오프를 설명하고 추천합니다. 기술 선택도 침묵 속에서 확정하지 않습니다.

사용자에게 결과가 가기 전에는, 작성에 참여하지 않은 별도 에이전트가 한 번 더 봅니다. 사용자가 말한 내용이 빠지거나 왜곡되지 않았는지, 문서가 그대로 실행 가능한지 확인합니다. 그래도 최종 승인은 항상 사용자에게 있습니다.

## `/leanforge:run` — 말이 아니라 증거로 검증하는 실행

`/leanforge:run`는 승인된 계약을 받아 구현을 진행합니다. 이 시점부터는 git 상태를 기준으로 작업을 통제합니다.

**스케줄링.** plan에는 명시적인 의존성 그래프가 들어 있습니다. `/leanforge:run`는 어떤 파일도 바꾸기 전에 이 그래프를 검증합니다. 순환, 끊어진 참조, 빠진 범위가 있으면 실행 중에 임기응변으로 메우지 않고 조기에 실패시킵니다. 서로 독립적인 task는 병렬로 실행되고, 각 구현자는 격리된 git worktree에서 작업합니다. 병합 전에는 브랜치가 실제로 전진했는지, diff가 선언된 범위만 건드렸는지 확인합니다. 구현자의 자기 보고만으로는 통과하지 않습니다.

**검증은 위험에 맞춥니다.** 단순 rename과 결제 경로 변경은 같은 무게로 다루지 않습니다. 위험한 task는 spec 일부와 raw diff를 기준으로 별도 리뷰를 받습니다. 이때 구현자의 요약은 리뷰어에게 넘기지 않습니다. 요약이 리뷰를 끌고 가는 일을 막기 위해서입니다.

**게이트는 끝까지 갑니다.**

- 병렬 wave가 끝날 때마다 병합된 base에서 프로젝트 검증 스위트를 실행합니다. task 사이의 충돌은 이때 처음 드러납니다.
- 완료 직전에는 전체 검증을 다시 돌리고 **런타임 smoke**를 추가합니다. spec에 서비스가 있다면 실제로 부팅되고 요청에 응답해야 합니다. *컴파일된다*와 *동작한다*는 다릅니다.
- 검증 명령이 단언에 도달하기 전에 죽었다면 통과로 추정하지 않습니다. 실패로 봅니다.

**막히면 멈춥니다.** task가 사용자 판단을 필요로 하면 답을 가정하지 않고 기다립니다.

**git은 사용자 소유입니다.** 기존 프로젝트에서는 feature branch에서 실행합니다. main은 직접 쓰지 않습니다. 최종 통합 방식도 자동으로 정하지 않습니다. 더러운 working tree나 push되지 않은 커밋이 있으면 시작 전에 멈춥니다.

모든 게이트를 통과하면 `/leanforge:run`는 프로젝트 하네스를 만들거나 갱신합니다. 그런 다음 전체 변경을 한 번 더 독립 리뷰하고, 설계 계약을 `.leanforge/` 아래에 보관합니다. 승인이나 아카이브 전에 실행이 끊기면 `.leanforge/run.json`이 복구 상태를 남깁니다. 다음 에이전트는 이를 보고 재개할지, 버릴지 명시적으로 선택할 수 있습니다.

## `/leanforge:set` — 기존 코드베이스 온보딩

`/leanforge:set`은 1회성 온보딩 명령입니다. task 실행기가 아닙니다. 먼저 코드베이스를 훑고, 코드만으로는 알 수 없는 것을 사용자에게 확인합니다. 예를 들어 코드 경로는 auth 체크가 실제로 무엇을 하는지는 보여주지만, 그것이 정책의 전부인지는 말해주지 않습니다.

질문은 위험도에 맞춰 조절됩니다. 코드와 이전 답에서 추론할 수 있고 틀려도 비용이 작은 것은 묻지 않습니다. 반대로 도메인 불변식, 보안 경계, 체크 뒤의 비즈니스 규칙처럼 틀리면 비싼 것은 사용자에게 확인합니다.

질문은 평이한 말로 도착합니다. *"이게 바뀌면 저것도 같이 바뀌어야 하나요?"* 같은 식입니다. 답은 다시 정확한 규칙으로 정리되어 하네스에 들어갑니다. 하네스 생성이 끝나면 커밋 여부는 사용자에게 남기고 종료합니다. 그 뒤의 작업은 `/leanforge:prime` → `/leanforge:run` 흐름으로 이어집니다.

## 한 사이클은 이렇게 돕니다

`/leanforge:prime` → `/leanforge:run`에는 사용자가 승인하는 지점이 두 번 있습니다. 계약을 승인할 때 한 번, 완료 결과를 승인할 때 한 번입니다. 그 사이의 실행은 계약 안에서 자율적으로 진행됩니다.

```
/leanforge:prime 입력 분해 → 열린 결정 정리 → 계약 작성
          → 독립 검토 → ▶ 사용자 승인

/leanforge:run   그래프 검증 → 병렬 wave 실행 → 통합 게이트
          → 런타임 smoke → 하네스 생성/갱신 → 최종 리뷰
          → ▶ 사용자 승인 → 계약 아카이브
```

`/leanforge:prime`가 `.leanforge/`에 남기는 것은 세 문서로 된 **설계 계약**입니다.

- **spec** — 무엇을 만들어야 하는지의 기준입니다. 동작 규칙, 불변식, API 표면, 엣지 케이스 처리, 통과해야 할 검증을 담습니다.
- **plan** — 어떻게 실행할지의 청사진입니다. task별 동작 계약과 `/leanforge:run`가 사용할 의존성 그래프를 담습니다.
- **handoff** — 세 문서의 관계, 충돌 시 우선순위, 어떤 단계도 넘을 수 없는 hard gate를 담습니다.

이 계약은 대화 기록 없이도 읽을 수 있어야 합니다. 미래의 에이전트가 그대로 보고 움직일 수 있게 쓰입니다. 코드에서 다시 추론할 수 없는 결정은 이유까지 본문에 남깁니다. 권위도 명확합니다. spec과 코드가 어긋나면 spec이 이깁니다. spec 자체가 틀려 보이면 에이전트가 혼자 고치지 않고 사용자에게 돌아옵니다.

`/leanforge:run`이 끝나면 계약은 `.leanforge/` 아래에 사이클별로 보관됩니다. 무엇을, 언제, 왜 결정했는지 남습니다. 진행 중인 작업은 `.leanforge/{handoff,spec,plan}.md`로 보이고, 중단된 실행은 `.leanforge/run.json`으로 보호됩니다.

Legacy 상태 호환성도 있습니다. Leanforge로 이름을 바꾸기 전의 저장소에는 `.dryforge/`가 남아 있을 수 있습니다. 현재 기준 디렉터리는 `.leanforge/`입니다. `.leanforge/`가 없고 `.dryforge/`만 있으며 active `run.json`, 루트 3문서, `worktrees/`가 없을 때만 `Run`/`Set`이 `.leanforge/`로 옮기고 `.leanforge/migration.json`을 기록합니다. 진행 중인 legacy run은 자동으로 옮기지 않습니다. 먼저 완료하거나, 재개하거나, 명시적으로 버려야 합니다.

## 프로젝트 하네스에 남는 것

```
your-project/
├── CLAUDE.md                  # Claude Code 진입점 — 정체성 + 작업 규칙
├── AGENTS.md                  # Codex 진입점 — 동일 내용
├── docs/
│   ├── architecture.md        # 구성: 컴포넌트, 흐름, 의존성
│   ├── business-rules.md      # 도메인 로직: 엔티티, 불변식, 엣지 케이스
│   ├── security.md            # 정책: 보호 자산, 접근, 감사
│   ├── standards.md           # 규칙: 하드 게이트, 관례, 경계
│   ├── engineering-notes.md   # 체득 지식: 함정, 메커니즘, 체크리스트
│   ├── operations.md          # 구동법: 셋업, 빌드, 배포
│   ├── contracts.md           # 외부 인터페이스 계약
│   └── tracking/
│       ├── status.md          # 전체 scope 대비 현재 위치
│       ├── decisions/         # 결정 기록 — 무엇을, 왜 선택했는가
│       └── findings.md        # 미해결로 알려진 문제
└── <module>/AGENTS.md         # 모듈별 범위, 경계, 불변식
```

- **세션이 바뀌어도 이어집니다.** 새 대화는 하네스를 먼저 읽고 아키텍처, 규칙, 결정 이유를 가진 상태에서 시작합니다. 같은 설명을 매번 반복할 필요가 없습니다.
- **이유가 같이 남습니다.** 무엇을 선택했는지만이 아니라 왜 그렇게 했는지도 남습니다. 한 번 정리한 논쟁을 다음 세션에서 다시 시작하지 않기 위해서입니다.
- **문서는 덧붙이기만 하지 않습니다.** 새 작업이 기존 설명을 무효화하면 그 설명도 고칩니다. 하네스는 현실을 따라가야 하고, 틀렸다면 수정되어야 합니다.
- **락인이 없습니다.** 결과물은 표준 `CLAUDE.md`, `AGENTS.md`, 일반 `docs/`입니다. 생성된 문서는 Leanforge 전용 포맷에 갇히지 않습니다. 내일 Leanforge를 지워도 프로젝트 자산은 남습니다.

---

## 다른 도구와 무엇이 다른가

계획을 세우거나 실행을 조율하는 도구는 많습니다. Leanforge의 차이는 무엇을 믿고 시작하느냐에 있습니다.

- **그냥 에이전트를 쓰는 경우와 비교하면** — 강한 모델도 기준점이 없으면 매 세션 프로젝트를 다시 추론합니다. 모델이 강할수록 더 그럴듯하게, 더 멀리 벗어날 수 있습니다. Leanforge는 매 세션 같은 결정과 같은 이유를 제공합니다.
- **스펙 생성기나 plan mode와 비교하면** — 그런 도구는 보통 사용자가 말한 내용을 정리합니다. Leanforge는 사용자가 말하지 않았지만 구현 전에 정해야 하는 것도 찾아냅니다. 들어온 문서도 그대로 포맷팅하지 않고, 충돌과 빈칸을 확인합니다.
- **처방형 워크플로와 비교하면** — 고정 규칙집은 모든 작업에 같은 절차 비용을 부과하고, 품질을 규칙 작성자의 예측 범위 안에 묶습니다. Leanforge는 계약과 최소한의 바닥만 고정하고, 판단은 모델이 하게 둡니다. 모델이 좋아지면 산출물도 같이 좋아질 여지를 남깁니다.
- **병렬 오케스트레이터와 비교하면** — 의도 정리 없이 병렬화하면 잘못된 결과를 더 빨리 만들 뿐입니다. Leanforge는 승인된 spec 아래에서만 병렬화하고, 게이트를 통과한 것만 합칩니다.
- **상태 대시보드와 비교하면** — 많은 도구는 실행 후 이벤트를 추적합니다. Leanforge는 결정을 추적합니다. 실행 로그는 무슨 일이 있었는지 알려주지만, 설계 계약은 다음 에이전트가 왜 그렇게 해야 하는지 알게 합니다.

## 철창이 아니라 나침반

- **천장이 아니라 바닥입니다.** Leanforge는 인터페이스 계약과 절차의 뼈대를 고정합니다. 결론은 모델이 내립니다. 규칙으로 모든 것을 처방하면 도구의 품질이 규칙 작성자의 예측력에 묶입니다. 바닥만 깔아 두면 더 나은 모델이 더 나은 결과를 낼 수 있습니다.
- **자율에는 경계가 있습니다.** 사용자가 의도를 승인하면, 에이전트는 그 경계 안에서 자유롭게 판단합니다. 자율은 승인된 의도를 실행하는 능력이지, 의도를 스스로 정하는 권한이 아닙니다.
- **모르면 묻습니다.** 의도에서 도출되지 않는 것은 진행하기 전에 사용자에게 돌아옵니다. 추측 한 번이 질문 한 번보다 비쌀 때가 많습니다.

## 플랫폼 노트

- **명시적으로 불러야 동작합니다.** core lifecycle은 자동으로 트리거되지 않습니다. 호출하지 않으면 아무것도 실행되지 않습니다.
- **소스는 하나, 플랫폼은 둘입니다.** Claude Code와 Codex 번들은 플랫폼 중립 `src/skills/` 트리에서 `claude/`와 `codex/plugin/`으로 빌드됩니다. 릴리스 전에는 plugin version parity와 shared reference file parity를 확인합니다.
- **핵심 표면.** `Prime`, `Run`, `Set`이 완전한 core lifecycle을 구성합니다. Codex 번들에는 선택형 `Run TDD` wrapper도 포함됩니다.
- **필수 요구 사항.** `git`, 그리고 Claude Code 또는 Codex가 필요합니다.

---

## 언제 쓰지 않는 게 좋은가

- **한 줄 수정에는 과합니다.** Leanforge는 재작업을 줄이기 위해 elicitation과 검증 비용을 먼저 냅니다. 기능 개발이나 프로젝트 단위 작업에서는 이 비용이 회수되지만, 단순 오타 수정에는 맞지 않습니다.
- **git이 필요합니다.** 실행 규율이 branch와 worktree 위에 세워져 있습니다.
- 질문에 답하는 시간은 실행에서 회수됩니다. 이후 사이클에서도 다시 회수됩니다. 에이전트가 사용자에게 다시 묻는 대신 하네스를 읽기 때문입니다.
